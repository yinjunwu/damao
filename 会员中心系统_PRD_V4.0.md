## 一、产品概述

### 1.1 产品定位

会员中心系统是面向注册用户的个人中心管理平台，提供会员信息管理、权益查看、任务参与、购买记录查询以及发票申请等一站式服务功能。系统采用现代化的前端技术栈构建，以清晰的信息架构和友好的交互体验为核心设计目标。

### 1.2 目标用户

1. 平台注册用户（含免费注册用户和付费会员）
2. 需要进行会员升级或购买虚拟商品的用户
3. 需要对已消费金额申请发票的企业/个人用户

### 1.3 技术架构

| 技术项 | 方案/版本 |
|:---|:---|
| 前端框架 | React 19 |
| 开发语言 | TypeScript |
| 构建工具 | Vite |
| 样式方案 | Tailwind CSS |
| 包管理器 | npm |
| 运行环境 | Node.js 22.12.0+，现代浏览器(Chrome/Firefox/Safari/Edge) |

## 三、页面布局设计

### 3.1 整体布局结构

页面采用最大宽度 **1400px** 的居中容器，整体结构分为三个层次：

1. **顶部Header区**：包含"会员中心"标题、副标题描述及"会员体系介绍"按钮
2. **主内容区（双栏）**：
   - 左侧固定宽度 **400px** 的会员信息面板（MemberInfoPanel）
   - 右侧自适应宽度的 Tab 内容区域
3. **右侧内容区内部结构**：顶部 Tab 导航栏 + 底部内容展示区

### 3.2 Tab 导航顺序

7 个一级 Tab 从左到右依次排列：

```
资格认证 | 权益券 | 兑换码 | 优惠券 | 会员升级 | 任务中心 | 购买记录
```

**交互规范**：
- 激活态 Tab 文字呈 `indigo-600` 色，底部带有 `indigo-600` 圆角下划线指示器
- 非激活态文字为灰色，hover 时变深灰色
- 默认激活"任务中心"Tab

## 六、购买记录 / 订单管理（OrderManagement）

### 6.1 页面概述

购买记录页面提供订单列表的查询、筛选、多选操作以及批量申请开票功能。该页面是系统的**核心业务页面**。

### 6.2 工具栏设计（V3.0 重构，V4.0 继承）

工具栏采用**单行左右分栏布局**，将筛选条件和操作按钮整合为一条紧凑的工具栏。

#### 6.2.1 左侧 — 筛选条件区

| 控件 | 类型 | 规格 | 说明 |
|:---|:---|:---|:---|
| 订单状态 | 下拉选择框 | rounded-lg 圆角 | 全部 / 已完成 / 待支付 / 已取消 |
| 关键词搜索 | 文本输入框 | 带内嵌搜索图标 + 清除按钮 | placeholder: "搜索订单号/商品名..." |

#### 6.2.2 右侧 — 操作按钮区

所有按钮采用**轻量级图标风格**（p-1.5 内边距），hover 时才高亮：

| 按钮 | 显示时机 | 样式 | 说明 |
|:---|:---:|:---|:---|
| 开票(N) | 选中订单时 | bg-indigo-50 text-indigo-700 小型标签 | 金额不足50元时禁用 |
| 刷新 | 始终显示 | 纯图标按钮 | 点击触发刷新 |
| 导出 | 始终显示 | 纯图标按钮 | 导出当前筛选结果 |
| 重置 | 有筛选条件时 | 小型文本按钮 | 仅在有筛选条件时出现 |

### 6.3 订单表格列定义

表格共 **6 列**：

| 序号 | 列名 | 字段Key | 对齐方式 | 说明 |
|:---:|:---|:---|:---:|:---|
| 1 | 复选框 | checkbox | 居左 | 仅已完成订单可勾选 |
| 2 | 订单号 | orderNo | 居左 | 附带复制按钮 |
| 3 | 商品名称 | productName | 居左 | — |
| 4 | 订单状态 | status | 居左 | 已完成(绿) / 待支付(橙) / 已取消(灰) |
| 5 | 实付金额 | amount | 居左 | 红色加粗，带前缀 |
| 6 | 创建时间 | createTime | 居左 | 格式: YYYY/MM/DD HH:mm |

### 6.4 多选与开票逻辑

- 仅"已完成"状态的订单可被勾选
- 最低开票金额：50 元人民币
- 底部选中信息栏显示已选数量和合计金额

### 6.5 Props 数据传递（V4.0 新增）

OrderManagement 组件新增以下 props 从 MemberCenter 透传至 InvoiceDialog：

```typescript
interface OrderManagementProps {
  orders: OrderItem[]
```

### 7.2 认证信息模型（V4.0 新增）

```typescript
type CertificationStatus = 'none' | 'personal' | 'enterprise'
type MemberLevel = '普通会员' | '专业会员' | '企业会员'

interface CertificationInfo {
  status: CertificationStatus
  personalName?: string      // 个人认证姓名
  personalIdNo?: string      // 个人身份证号
  companyName?: string       // 企业认证公司名
  taxNo?: string             // 企业税号
}
```

### 7.3 表单字段详解

#### 7.3.1 可开票金额摘要卡

indigo 浅色背景卡片，显示可开票金额和金额明细。

#### 7.3.2 申请开票金额 *必填

number 输入框（带前缀），最小值50，最大值为选中订单总额，精度两位小数。

#### 7.3.3 费用项名称 *必填

两级联动下拉选择框（并排布局）：一级分类 + 二级子类。

#### 7.3.4 抬头信息 *必填（V4.0 全面重构）

根据会员认证类型自动展示对应的实名认证信息，**只读不可修改**：

| 认证状态 | 会员类型 | 展示样式 | 展示内容示例 | 底部说明 |
|:---|:---|:---|:---|:---|
| personal 已认证 | 普通/专业会员 | 浅色只读文本框（indigo边框） | 张三（110101199001011234） | 个人会员抬头，已绑定实名认证信息 |
| enterprise 已认证 | 企业会员 | 浅色只读文本框（indigo边框） | 某某科技有限公司（税号：91110000XXXXXXXXXX） | 企业会员抬头，已绑定企业认证信息 |
| none 未认证 | — | 虚线框可点击区域（amber色）+ 盾牌图标 + 「前往实名认证」 | — | 完成实名认证后可申请发票 |

未认证时的交互行为：
- 点击虚线框触发 `onGoToCert()` 回调跳转实名认证
- 提交按钮禁用，无法提交申请

#### 7.3.5 发票类型 *必填（V4.0 重构）

仅保留两个选项：**增值税普通发票**（默认）、**增值税专用发票**

选择不同发票类型时表单动态变化：

**增值税普通发票**：
- 仅展示基础必填字段
- 显示蓝色提示卡片：普通发票适用范围说明，不可抵扣进项税额
- 不展示专票额外字段

**增值税专用发票**：
- 基础字段 + 黄色提示卡片 + **4个额外必填字段**

| 字段 | Key | 占位符示例 | 布局 |
|:---|:---|:---|:---|
| 注册地址 | registeredAddress | 请输入企业注册地址 | 全宽 |
| 注册电话 | registeredPhone | 如：010-88886666 | 左半（两列并排） |
| 开户银行 | bankName | 如：招商银行北京望京支行 | 右半（两列并排） |
| 银行账号 | bankAccount | 请输入银行账号 | 全宽 |

#### 7.3.6 发票接收邮箱（V4.0 重构）

移除 Checkbox 逻辑，直接填写邮箱地址：

- email 类型输入框，始终可见
- 标签：「* 发票接收邮箱」（必填）
- placeholder："请输入接收发票的邮箱地址"
- 说明文字：「发票将通过邮件发送到该邮箱地址」

**邮箱记忆机制**：

| 操作 | 行为 |
|:---|:---|
| 打开弹窗 | 从 localStorage 读取上次保存的邮箱并回填 |
| 提交成功 | 将当前邮箱存入 localStorage 作为下次默认值 |
| 存储Key | `invoice_last_email` |
| 无历史记录 | 邮箱框为空 |

#### 7.3.7 底部操作按钮

| 按钮 | 启用条件 |
|:---|:---|
| 取消 | 始终可用 |
| 提交申请 | 金额>=50 且已认证 且邮箱不为空 且（专票时4个字段均已填） |
| 提交中 | 文案变为「提交中...」，禁止重复点击 |

### 7.4 提交处理逻辑（V4.0 新增）

不再使用 mailto 协议调用客户端邮件，改为通过后端 API 异步处理。

#### 接口规范

| 属性 | 规格值 |
|:---|:---|
| 接口路径 | POST /api/invoice/apply |
| Content-Type | application/json |
| 目标收件人 | business@mentx.com |

#### 请求体结构

```json
{
  "amount": "99.00",
  "feeItem": "信息技术服务",
  "feeSubType": "会员订阅",
  "invoiceType": "增值税专用发票",
  "payerDisplay": "某某科技有限公司（税号：91110000XXXXXXXXXX）",
  "registeredAddress": "北京市朝阳区望京科技园区A座8层",
  "registeredPhone": "010-88886666",
  "bankName": "招商银行北京望京支行",
  "bankAccount": "1100 8888 6666 1234",
  "receiveEmail": "user@example.com",
  "toEmail": "business@mentx.com"
}
```

#### 后端职责

接收发票数据 → 组装邮件内容 → 发送到 business@mentx.com

#### 前端容错

开发环境若后端未部署，catch 分支仍提示成功并关闭弹窗。生产环境必须部署后端接口。

## 九、关键交互流程

### 9.1 批量申请开票流程（V4.0）

1. 进入「购买记录」Tab → （可选）筛选/搜索订单
2. 勾选已完成订单 → 工具栏显示「开票(N)」按钮
3. 合计 >= 50 后开票按钮可用 → 点击弹出 InvoiceDialog
4. 弹窗根据 certInfo.status 自动展示对应抬头信息（个人/企业/未认证）
5. 选择发票类型（普通/专用），专用时补充4个字段
6. 填写接收邮箱（自动回填历史邮箱）
7. 点击「提交申请」→ POST /api/invoice/apply → 后端发送邮件到 business@mentx.com
8. 成功后邮箱存入 localStorage → 弹窗关闭

### 9.2 抬头信息展示逻辑（V4.0 新增）

- **personal**：展示姓名+身份证号（只读）
- **enterprise**：展示公司名+税号（只读）
- **none**：显示「前往实名认证」引导链接，提交按钮禁用
